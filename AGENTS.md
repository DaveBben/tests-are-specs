# AGENTS.md

Guidance for AI coding agents (Claude Code, Codex, OpenCode, and others) working in this repository.

## What this repo is

This repo is the `spec-monkey` skills: eight portable skills for spec-driven development. Seven are the phases of the flow â€” `grounding-specs`, `shaping-specs`, `reviewing-design`, `writing-specs`, `reviewing-specs`, `implementing-specs`, and `auditing-specs` â€” and `running-lifecycle` drives them end to end. Each skill is plain Markdown: a `SKILL.md` with YAML frontmatter and an instruction body, plus a reference file or two. There is no build step and nothing to compile. A skill has to run in any harness that can load Markdown skill instructions, so keep the wording harness-neutral. Name specific harnesses (Claude Code, Codex, OpenCode) only as examples.

## Key files

- `plugins/spec-monkey/skills/<name>/SKILL.md`: the eight skills. Frontmatter (`name`, `version`, `description`, `license`, `compatibility`, `allowed-tools`) followed by the instruction body. **These are the source of truth for behavior.**
- `plugins/spec-monkey/skills/grounding-specs/references/`: the project-spec format (`project-template.md`) and the architecture interview (`interview-questions.md`).
- `plugins/spec-monkey/skills/shaping-specs/references/`: the thinking interview (`interview-questions.md` â€” the ask, orientation, the five risk lenses, and the approach comparison), the design.md format (`design-template.md` â€” shaping's sole output, defined here), and the brief/PRD fast path (`ingesting-briefs.md` â€” ingest an existing document and interview only the gaps; shared, `writing-specs` points at it too).
- `plugins/spec-monkey/skills/writing-specs/references/`: the `spec.md` + `detail/contract.md` format (`spec-template.md`, composed here from the design) and the contract interview (`interview-questions.md`).
- `plugins/spec-monkey/skills/reviewing-design/references/design-review-rubric.md`: the rubric the **design** reviewer works through (approach soundness, code grounding, risk-lens coverage, undefended decisions, verification-strategy credibility, plus the project-spec checks). reviewing-design gates the design before a contract exists.
- `plugins/spec-monkey/skills/reviewing-specs/references/review-rubric.md`: the rubric the **contract** reviewer works through (the self-consistency sweep â€” absorbed from writing-specs â€” plus verification trustworthiness and the altitude gate). It judges `spec.md` + `detail/contract.md`, not the approach.
- `plugins/spec-monkey/skills/auditing-specs/references/`: the audit checklist (`audit-rubric.md`) and the opt-in fix-and-re-audit loop (`remediation.md`, loaded only when the user asks to fix).
- `plugins/spec-monkey/skills/implementing-specs/references/`: the build detail, each loaded on demand â€” `build-discipline.md` (the test-first loop, the faked-done anti-patterns, the self-review rubric), `debugging.md` (the root-cause escape hatch for a slice that won't go green), `build-workspace.md` (the isolated branch/worktree and the finishing decision), `subagent-mode.md` (the opt-in subagent-orchestrated build), and the three ready-to-use dispatch prompts it drives â€” `implementer-prompt.md`, `reviewer-prompt.md`, `fixer-prompt.md` (fill the bracketed slots and dispatch; they never paste FR/SC text, they point at `contract.md`). If you change the subagent loop, change the prompt files and `subagent-mode.md` together.
- `plugins/spec-monkey/skills/running-lifecycle/SKILL.md`: the orchestrator; it invokes the seven phase skills and has no reference file.
- `docs/testing-skill-changes.md`: the maintainer method for testing a skill-behavior change. Not shipped as a skill.
- `docs/interop.md`: how spec-monkey composes with superpowers (who owns which phase, and the `contract.md â†’ plan` seam). For humans.
- `tools/`: optional spec tooling outside the portable skills â€” `spec-lint.py` (mechanical parse-contract checks: placeholders, `FR`â†”`SC` pairing, ID uniqueness, dangling `INV-NNN`, `parent` resolution, status, `profile` value, and a missing gate record on an approved spec; plus a `--status` roll-up mode that tabulates every spec's kind/profile/status/gate) and its README. Stdlib Python, CI-runnable. No skill depends on it.
- `tests/`: control-first behavioral pressure scenarios for the discipline-critical skills (gate integrity, faked-done, HOW-leak, interview shortcut, brief ingestion, invariant break), plus `harness.md` (the reproducible control-vs-skill runner protocol) and `RESULTS.md` (genuine recorded runs â€” real observed output, never asserted or fabricated). Scenario inputs and pass/fail signals, not baked-in transcripts â€” the maintainer method in `docs/testing-skill-changes.md` runs them. Outside the plugin; no skill depends on it.
- `hooks/`: the optional multi-harness session bootstrap that auto-starts `running-lifecycle` â€” `session-start.sh` (sniffs env vars and emits the router line in the JSON shape the current harness consumes: Cursor `additional_context`, Claude Code `hookSpecificOutput.additionalContext`, SDK-standard `additionalContext` for the rest), `install-hook.sh` (the one-command, idempotent installer for the Claude Code settings.json path), and `hooks-cursor.json` (Cursor's hook config, referenced by the Cursor adapter). Outside the plugin, off by default, so the plugin stays hook-free.
- `docs/harness-tools/<harness>.md`: the per-harness tool mappings. Skills name actions ("dispatch a subagent," "run the spec linter"); these files translate that vocabulary into each harness's real tool names. The README there lists the action vocabulary every mapping must cover. Outside the skills on purpose â€” adding a harness adds a file here, never a skill edit.
- `docs/porting-to-a-new-harness.md`: the adapter-layer architecture (harness-agnostic skills / per-harness tool mapping / optional per-harness bootstrap), the capability table (what degrades vs. what's hard-required), and the steps to add a harness.
- `README.md`: for humans. Install steps, the eight-skill table, and the design principles.
- `.claude-plugin/marketplace.json`: the `spec-monkey` marketplace entry. It makes `/plugin marketplace add` work, and it lists each skill so `npx skills` groups them under the plugin.
- `plugins/spec-monkey/.claude-plugin/plugin.json`: the Claude Code plugin manifest.
- `.codex-plugin/plugin.json` and `.cursor-plugin/plugin.json`: the per-harness adapter manifests at the repo root, following superpowers' shapes. Codex points at the skills and declares an empty `hooks` object (Codex surfaces skills natively, no session hook); Cursor points at the skills and at `hooks/hooks-cursor.json` (installing the Cursor plugin is the opt-in for the bootstrap). Their `version` fields carry the package version â€” bump them with `plugin.json` and `marketplace.json`.

## The maintenance contract

Behavior lives in the skills; the README and the manifests describe them. Change one, change the others in the same commit.

- **Skill set:** the plugin ships eight skills. If you add, remove, or rename one, update the README's eight-skill table and prose, the `skills` array in `marketplace.json`, and the plugin and marketplace descriptions together. A skill's `name` must match its directory name. The two review skills split by target: `reviewing-design` judges the design/approach (`detail/design.md` and project specs), `reviewing-specs` judges the contract (`spec.md` + `detail/contract.md`) â€” keep their scopes disjoint.
- **Handoffs:** each skill's `## Next step` section names the next skill by directory name, and `running-lifecycle` names all seven phases. If you add, remove, or rename a skill, update every `## Next step` line and `running-lifecycle` that names it, in the same commit.
- **Harness-specific glue lives outside the skills.** A skill declares intent with `compatibility` and `metadata`, never a hard dependency. Anything that needs a hook or a subagent stays optional and external: the session bootstrap lives in `hooks/` (outside the plugin, off by default); the orchestrated build and the fix-and-re-audit loop are capability-gated opt-in modes with a portable default path. **Skill bodies name actions, never tools**: "dispatch a subagent," "open a new session with an empty context," "invoke a skill," "run the spec linter" â€” never `Task`, `TodoWrite`, a `subagent_type` string, slash-command syntax, or a `.claude/` path. The translation from action to real tool is per-harness and lives in `docs/harness-tools/<harness>.md`; if an edit seems to need a tool name in a skill, the edit belongs in a mapping file instead. The architecture and the steps to add a harness are in `docs/porting-to-a-new-harness.md`.
- **Version â€” three numbers, three questions.** spec-monkey carries three independent `version`-like fields on purpose. Each answers a different question, so they do not, and should not, march in lockstep:

  | Field | Where | Answers | Bump when |
  |---|---|---|---|
  | **Package version** | `plugin.json`, the `marketplace.json` plugin entry, **and** the per-harness adapter manifests (all must match) | "Which release of the whole plugin is this?" | any shipped change to the plugin |
  | **Skill version** | each `SKILL.md` frontmatter `version` | "Which revision of *this one skill's* behavior is this?" | you change that skill's instructions |
  | **Spec format version** | `spec_monkey:` in a spec's frontmatter | "Which version of the document *schema* was this spec written against?" | the template/parse-contract changes (rare) |

  The first is about the release, the second about a skill, the third about user data. That last one is why a spec you wrote against an earlier schema still reads `spec_monkey: "1.4.0"` even though the plugin is at 2.0.0 and the current schema is 1.7.0 â€” the spec records the schema it was authored against, not the plugin's, and it stays frozen at that number as the plugin moves on (the linter reads old and new alike). Package and skill versions are the plugin's; the format version is the spec's.

  Bump rules: the package version lives in places that must match â€” the `version` in the `marketplace.json` plugin entry, the `version` in `plugin.json` (Claude Code treats this one as authoritative), and the `version` in each per-harness adapter manifest (`.codex-plugin/plugin.json`, `.cursor-plugin/plugin.json`); bump them all together. Bump a skill's own `version` when you change that skill's behavior. Bump `spec_monkey` (in both templates and the linter's accepted set) only when the spec schema itself changes.
- **Portability:** keep install and usage language harness-neutral. Harness-specific hints (`model`, `effort`, `argument-hint`) go under `metadata:`, where a harness that doesn't read them just skips them. Never require a subagent, a hook, or context forking from inside a skill. Only a handful of harnesses have those, so anything that depends on one stops being portable.
- **Descriptions:** `name` and `description` load at startup for every skill, so every token has to earn its place. Lead with the trigger, keep the description near 130 tokens or under, and push the detail into the body and the reference files.
- **One home for each spec format:** the work-item spec spans two format files, split by author. `shaping-specs` owns `detail/design.md` (`shaping-specs/references/design-template.md`, its sole output); `writing-specs` owns `spec.md` + `detail/contract.md` (`writing-specs/references/spec-template.md`, composed from the design it reads as input). The project-spec format lives in `grounding-specs/references/project-template.md`. The rubric and the skills point at them; they don't restate them. Both share the `spec_monkey` version and the lifecycle convention; the ID conventions split by document (`FR-`/`SC-` in work-item specs, `INV-` in the project spec). If any of those change, update both templates, the rubric, and any README mention in one change. The work-item template also defines the `profile` field (`full`, the default, or `light` â€” the trivial lane's single-file reduced schema; see "THE LIGHT PROFILE" in `spec-template.md`). `tools/spec-lint.py` encodes the machine-readable half of the parse contract (frontmatter scalars incl. `profile`, the `FR`â†”`SC`/`INV` join keys, the status values, and the gate record â€” `approved_by` present once status reaches `approved`) and pair-checks a light spec in its `spec.md` rather than in `detail/contract.md`; if you change one of those conventions, update the linter and re-run it against `examples/` so it still passes clean.
- **WHAT and WHEN, never HOW:** a spec says what to build and when, not which file or symbol to edit. Don't let build detail leak into a skill or the template. The one exception is the verification commands, which belong in the spec.
- **Verification is two obligations, synced across three skills.** The contract's *Verification approach & commands* splits into *Artifacts to author* (tests that must EXIST in the diff, NEEDS-INFRA ones included) and *Gates to pass* (commands that must RUN). This is what stops a named test being deferred to "run later": `implementing-specs` authors every artifact, `auditing-specs` check 2 FAILs on a missing artifact (not an open item), and `reviewing-specs` requires a per-boundary artifact line. Change the structure in `spec-template.md` and you change all three together.

## Editing a skill

- Keep the YAML frontmatter valid, in both formatting and indentation.
- The body under the frontmatter is the product. Edit it like a careful instruction document, not code.
- Keep reference files one level deep from `SKILL.md`, and move heavy detail down into them so the always-loaded body stays lean.

## Changing a skill's behavior

A skill edit is a behavior change until you have watched it change behavior. Wording that reads clearly to you can move an agent not at all. Before you commit an edit meant to change what an agent does (not a typo, a link, or a ragged line), test it against a no-guidance control. The method is in [`docs/testing-skill-changes.md`](docs/testing-skill-changes.md).

- **Control first.** Run the task with a fresh agent that does not have the skill. If it already does the right thing, the wording has nothing to fix; don't add it.
- **Match the form to the failure.** A shaping problem (the spec comes out with HOW in it) wants a positive recipe. A discipline problem (the agent skips the interview when the user is impatient) wants a prohibition plus the excuse it answers. The two forms backfire when swapped; don't reach for a prohibition by default.
- **This is a maintainer method, not a skill.** Don't add a testing skill to the shipped eight. It would trip the maintenance contract above and spend description tokens every session for something only maintainers run.
